What is Database?
It is collections of Data.

Example:
www.hdfc.com --> Public subnet (Webserver: static content = Home page)
  Login page --> [username & password] Private subnet (Appserver & Database) [Directly, we cannot login to the private subnet; we need to come to the public subnet from there we can be redirected to the private subnet for a secured connection]
So, whatever the transactions we are doing this will get updated into the Database for future reference purpose.

Under Database, we will be able to perform multiple activities like Updation / Deletion / Insert / Table Creation / Alter / Drop etc.,

Types of Database Management System:
1. Structured DB --> This will follow certain format / set of rules to store or retrieve the datas; in other words it is called as [schema based storage] ex: oracle, mysql etc.,
2. Semi-Structured --> It is a form of structured data, but doesn't follow/obey any tabular structured of data models associated with RDS or other forms of data tables ex: NOSQL DB, Email, EDI (Electronic data interchange) etc.,
3. Unstructured --> This support either [schema based or non-schema based], here we will do manual upload and analysis, ex: hadoop, ETL tools (exract, transform, load) tabuleau, data stage etc.,

Different types of Schema:
Physical --> This is going to mention where our data's are going to get saved
Logical --> This will mention how our data's are going to get represented

DB will default have only read/write permissions.

RDS: (Relational Database)
In real-time we will run DB under primary-replica basis [other words, we will have one for primary contact, if that gone down, the backup db will come into actions], this is called as Business Contunity Plan (BCP). 
We will bring this under grouping, where Master/Primary DB will be running under one subnet; and Replica will be in a different subnet, this is how we can bring high availability for our DB as well. Only by grouping we can bring sync inbetween Primary and Replica DB.

So, here we are going to create grouping for subnet inorder to bring sync inbetween our DB's

2. Select Databases - create database [switch to original interface - for better hands-on experience] - select Mysql [only it is eligible for free-tier] - select dev/test under use case - [it is must to attach the license model for the DB, except oracle; Amazon has license for all the other 5 Db's] - select the recent version of mysql - select t2.micro under Db instance class - [we are going to manually create the replica zone, hence give no under Multi AZ deployment] - Disable Storage autoscaling [Extra cost] - Provide a name, master username, master password & confirm password [ex: greensmaster; admin; admin123 & admin123] - select default VPC - select our subnet group - [as db is private one hence give no for public accessibility] - under AZ select one AZ [ex: us-east-2a] - attach existing security group [make sure all the ports are open for this testing purpose] - give a name for your databse [ex: greens] - [no need for IAM, kerbaros authentication, backup {disable copy tage to snapshots}, monitoring, logs, disable auto minor version upgrade {as it is only for testing purpose} mantainence window, deletion protection --> as all this are going to bring extra charges for us] - create database.

3. Create an linux Instance - bring 30gb for storage - open all ports [If you are not opening all ports, make sure you enabled mysql protocol under SG].

4. Login into Instance - Change as root user [sudo su -] - [by default mysql will not be available inside the instance] - install mysql [yum install mysql -y]

5. Now, we are going to catalog the ec2 instance with db server using endpoint - copy the endpoint from the db [ex: greensmaster.crtwtqq2zk6a.us-east-2.rds.amazonaws.com] - go to the instance - give command (mysql -h [endpoint] -P 3306 -u admin -p) {-h = host; -P = mysql port number; -u = username; -p = password} if facing error like (error 1049: unknown database) then use command, (mysql -h [endpoint] -u admin -p) [port number is not needed here]; enter - it will prompt for the db password which we generated already - provide the password - this will comes inside our mysql db.

6. Now, we are going to create queries [get the queries from another document (RDS Queries)]

7. Now, we are going to take replica; select the database - actions - Create read replica - provide DB instance identifier with a name [ex: greensreplica] - disable storage autoscaling [as it is going to charge as extra] - select different AZ for replica [for high availability] - disable maintanence for minor updates - create read replica.

Once we created the replice, now the Role will be changed from Instance to Primary.

Common things between Primary and Replica:
1. Credentials: admin / admin123
2. Endpoint for both Primary and Replica will be same
[primary: greensmaster.crtwtqq2zk6a.us-east-2.rds.amazonaws.com
 replica:   greensreplica.crtwtqq2zk6a.us-east-2.rds.amazonaws.com]

when primary will be on healthy state, this will perform: Read/Write actions --> and replica will perform: Read actions --> because both the DB's are under sync
when Primary went down --> then replica will take the primary role and it will perform both the Read/Write actions.

Try Delete the Primary Database: click actions - delete - [make sure to diasble create snapshot] - enable the acknowledge - type delete me - delete

when the activity is performed by 1 single endpoint for both the Read/Write operation then the performance will be normal.
