What is CI?
	Continuous Integration, where developers frequently merge (or integrate) their code changes into a shared code repository, often several times a day. Each time new code is added, it automatically triggers a series of tests and builds. The goal is to ensure that every small change works well with the existing codebase and doesn’t break anything. If there's an issue, developers are notified, so they can quickly fix it.
Example:
--------
	Imagine a team is building an online store website. Developer A works on the shopping cart, and Developer B works on the checkout process. In a CI setup, as soon as either developer updates their code in the shared repository, it triggers automated tests. If Developer A accidentally adds a bug, the CI system will flag it. This way, problems are detected early, before they reach customers.

What is CD?
	After CI, we have Continuous Deployment or Continuous Delivery. Both involve taking the changes tested in CI and automatically deploying them to production (CD) or preparing them for release (Delivery).

 > Continuous Delivery means that the software is always in a "ready-to-release" state, but the release is manually triggered.
 > Continuous Deployment takes it a step further by automatically releasing every change that passes tests to production.

Example:
--------
	Let’s say a mobile app development team uses CD. Once the app passes all the tests, it’s either set for manual approval (Continuous Delivery) or directly released on the app store (Continuous Deployment). This let users receive app updates frequently without waiting for major releases.

In general, we have in CI part (Build & Test) & if the test passes this moves to the CD section [Build Docker image, push it to a repository, add it to Kubernetes file, adn then deploy it]; with Jenkins we can automate the entire process, this can be achieved through configuring Pipelines.

MASTER-SLAVE:
=============
	In a Jenkins master-slave setup, there is one central Jenkins master server that manages and coordinates the build jobs, and multiple Jenkins slave nodes that execute the build jobs. This setup is useful for distributing the workload across multiple machines and achieving better performance and scalability.
Here's how a Jenkins master-slave setup works:
	Jenkins Master: The Jenkins master is the central server that manages the entire Jenkins environment. It hosts the web interface where users can configure jobs, view build results, and manage the Jenkins system. The master is responsible for scheduling build jobs, distributing them to the slave nodes, and monitoring their execution.
	Jenkins Slave Nodes: Slave nodes are individual machines or virtual instances that are configured to offload the actual execution of build jobs from the master. These slave nodes connect to the Jenkins master over the network and await instructions to execute build jobs. Slave nodes can be configured with different operating systems, software environments, and hardware configurations to cater to diverse build requirements.
	Jenkins also has the capability of distributing build jobs across a large number of systems, which is known as Jenkins distributed builds.

[*FYI: there is something called Job landing path: (cd /var/lib/jenkins/workspace) by going inside this path it will show the entire jobs that we built already, try this on the master machine, move inside the path and give (ls) command]

Jenkins Master is the primary Jenkins server and is responsible for the following tasks:

    - It distributes the builds among the numerous slaves for execution.
    - It organises the build projects.
    - It keeps an eye on the slaves at all times.
    - Master can also run build jobs directly if necessary.
    - It keeps track of the build outcomes and shows them.

Jenkins Slave runs on a remote machine. A slave is responsible for the following tasks:

    - Slaves can be operated on a number of different operating systems.
    - It responds to the Jenkins Master's demands. 
    - we can always arrange the project to run on a certain sort of slave computer.

Multi-Node Demo:
================
1. How to execute if I am having multiple machines, and I need to execute the tasks, that is where pipeline comes where we can write a Groovy script and try executing the same

2. Let's try using a sample pipeline script, create a New item, give name (Multi-Node-Demo) choose pipeline and add the script

pipeline {
    agent none
    stages {
        stage('Parallel Job on Multiple Nodes') {
            parallel {
                stage('Run on Master') {
                    agent { label 'master' }
                    steps {
                        echo "Running on Master"
                        sh 'hostname'
                    }
                }
                stage('Run on Slave1') {
                    agent { label 'sample' }
                    steps {
                        echo "Running on Slave 1"
                        sh 'hostname'
                    }
                }
            }
        }
    }
}

[*** Do remember to rename the labels properly, and when comes to master we don't have name label yet, to add one, click on manage jenkins - nodes - click on the settings icon (configure icon) - we can see the Labels there add the name (master) - apply & save

3. Now execute the Pipeline and we can see this got executed on both the machines simultaneously, once the pipeline got successful we are going to check for the output from the console output area.
