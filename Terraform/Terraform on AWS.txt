Terraform: An Overview
======================
	Terraform is an open-source Infrastructure as Code (IaC) tool developed by HashiCorp. It allows users to define, provision, and manage cloud infrastructure using a declarative configuration language called HCL (HashiCorp Configuration Language).

How Terraform Works in Real-Time:
---------------------------------
Write Configuration: Define infrastructure using .tf files.
Initialize (terraform init): Prepares Terraform by downloading necessary provider plugins.
Plan (terraform plan): Shows what changes will be made before applying them.
Apply (terraform apply): Creates, updates, or destroys resources.
State Management (terraform state): Keeps track of deployed infrastructure.
Destroy (terraform destroy): Removes all managed resources.

Related Terminologies:
----------------------
Providers: Connect Terraform to cloud services (AWS, Azure, GCP).
Modules: Reusable Terraform configurations.
State File (terraform.tfstate): Stores infrastructure details.
Backend: Defines where Terraform state is stored (local, S3, etc.).
Variables (var.tf): Dynamic values for configurations.
Outputs (output.tf): Returns useful data after deployment.
Provisioners: Execute scripts on a created resource.
Workspaces: Isolate different environments (dev, staging, prod).

Benefits of Using Terraform in a Company:
-----------------------------------------
Infrastructure as Code (IaC): Automates infrastructure deployment.
Multi-Cloud Support: Works across AWS, Azure, GCP, and on-premises.
Version Control: Tracks infrastructure changes in Git.
Scalability: Easily provisions and modifies infrastructure.
Collaboration: Teams can work together using remote state storage.
Cost Optimization: Identifies unused resources to save costs.
Security & Compliance: Ensures consistent policies and governance.

Terraform - Infrastructure as Code (IaC)
	Terraform is an Infrastructure as Code (IaC) tool that helps developers and system administrators automate the provisioning and management of cloud resources. It supports multiple cloud providers like AWS, Azure, Google Cloud, and on-premises infrastructure.

1. How Terraform Works:
-----------------------
Terraform follows a declarative approach, meaning you define what infrastructure should look like instead of specifying how to create it.

Terraform Workflow:
-------------------
1. Write Configuration (.tf files)

 > Define the infrastructure using the HashiCorp Configuration Language (HCL).

Example:
provider "aws" {
  region = "us-east-1"
}

resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
}

2. Initialize Terraform (terraform init)

 > Downloads the required provider plugins (like AWS, Azure, etc.).
 > Sets up the backend (local or remote) to store the state.

3. Plan Changes (terraform plan)

 > Shows what Terraform will do before applying changes.
Example Output:

+ aws_instance.web
    ami: "ami-0c55b159cbfafe1f0"
    instance_type: "t2.micro"

4. Apply Changes (terraform apply)

 > Deploys the infrastructure based on the plan.
 > You will be prompted to confirm changes:

Do you want to perform these actions? (yes/no)

5. State Management (terraform state)

 > Maintains a record of resources created and tracks changes.
 > Stored in a terraform.tfstate file.

6. Destroy Resources (terraform destroy)

 > Deletes all resources created by Terraform.
 > Useful for cleaning up environments.

2. Key Terraform Components
---------------------------
Providers
 > Providers act as a bridge between Terraform and cloud services.
 > Example providers: AWS, Azure, GCP, Kubernetes, GitHub.

Example:

provider "aws" {
  region = "us-east-1"
}

Resources
 > A resource defines a cloud service like EC2, S3, RDS, etc..
Example:

resource "aws_s3_bucket" "my_bucket" {
  bucket = "my-unique-bucket-name"
  acl    = "private"
}

Variables
 > Variables allow parameterized configurations.
Example:

variable "instance_type" {
  default = "t2.micro"
}

resource "aws_instance" "example" {
  instance_type = var.instance_type
}

Outputs
 > Outputs provide information after Terraform runs.
Example:

output "public_ip" {
  value = aws_instance.example.public_ip
}

Modules
 > Reusable Terraform configurations that can be shared across projects.
Example structure:

├── main.tf
├── variables.tf
├── outputs.tf
└── modules/
    ├── vpc/
    ├── ec2/
    └── rds/

Example module usage:

module "vpc" {
  source = "./modules/vpc"
}

State Management
 > Terraform stores infrastructure details in a state file (terraform.tfstate).
 > It helps track resource changes and prevents conflicts.

Types of Backends for State Storage
 > Local Backend (default, stores terraform.tfstate locally).
 > Remote Backend (S3, Azure Blob, etc.) for team collaboration.
Example remote backend configuration for AWS S3:

terraform {
  backend "s3" {
    bucket = "my-terraform-state"
    key    = "terraform.tfstate"
    region = "us-east-1"
  }
}

What is use of IaC?
	By running the same code, we can create the infrastructure repeatedly just by executing it. By doing like this, we can save our time and effort and this helps to reduce the error that may happen while creation.

Inside AWS also we have a tool called CloudFormation, that helps us to create the Infrastucture by running the code; but why do I need to use the Terraform code?
	If we are going to create using CloudFormation, this supports only when we use our cloud provider as AWS; but what if my cloud provider is Azure or GCP or something else, then using Terraform is the best option as they provide services for all providers.

Under Configuration Automation, we already gone through Ansible, Salt; now what is the use of Terraform?
	Terraform => Creates Infrastructure (Server, Network, Storage etc.,) --- acts like Civil Engg, who build the Infrastructure
	Ansible/Salt => Run the task inside the Infrastructure (Installing Packages, Running services etc.,) --- acts like Interior Designer, who works inside the Infrastructure.
So, There is need for both the tools is a must one.
